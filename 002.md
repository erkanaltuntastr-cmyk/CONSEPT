# Oakwood – Core Engine & First Product Strategy

## Strategic Decision (Locked)

Oakwood is built on **two inseparable pillars**:

1. **A flagship Fitness product (A)**
2. **A reusable Core Engine / Model (C)**

The fitness product is **not a side project**. It is the **proof vehicle** for the Core Engine.
The Core Engine is **not an abstract framework**. It is validated only through real products.

This dual strategy is intentional and non-negotiable.

---

## What Oakwood Actually Sells

Oakwood does **not** sell apps.
Oakwood sells a **proven behavioural tracking and interpretation model**.

* B2C sees: a calm, simple fitness app
* B2B sees: a reliable data & behaviour engine

The app builds trust.
The engine builds value.

---

## The Core Engine (Oakwood Core)

The Core Engine is the stable foundation reused across all products.

### Responsibilities (Must Not Change)

* Event-based micro logging
* Time-based aggregation (daily / weekly)
* Progress & streak logic
* Noise reduction and signal extraction
* Human-scale data formatting
* AI-ready structured outputs

### Explicit Non-Goals

* No UI logic
* No domain-specific assumptions
* No feature experimentation

If a requirement violates these rules, it does not belong in the Core.

---

## Domain Layer (Fitness – First Vertical)

The Fitness domain configures the Core without modifying it.

Examples:

* Activity types
* Intensity or effort indicators
* Recovery / rest signals

The domain layer may evolve.
The Core must remain stable.

---

## Product Layer (Fitness App)

The first Oakwood product is a **fitness tracking application**.

Purpose:

* Validate the Core Engine
* Generate early user trust
* Test pricing and behaviour

Design principles:

* Single-screen interaction
* Preset inputs
* 2–3 second logging
* Calm, non-addictive UX

---

## Growth Logic

1. Build Core + Fitness together
2. Prove daily usage and retention
3. Stabilise Core
4. Reuse Core for next vertical (Education)

No second vertical starts until Fitness demonstrates real usage.

---

## Business Implications

* Fitness app: low-cost, B2C trust builder
* Core Engine: high-value, B2B licensing asset
* White-label deployments possible only after Core stabilisation

---

## Decision Filter

Any future idea must answer:

1. Does this strengthen the Core?
2. Does this help validate the Fitness product?

If the answer is no to both, the idea is rejected.
